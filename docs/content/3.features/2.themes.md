Das Theme-System lebt im Layer `themes` (die Tables besitzt `system`, die Admin-Routen registriert `admin` — Besitzverhältnisse siehe CONCEPT.md A14). Vollständiges Konzept inkl. bewusster Ablehnungen: `docs/THEMES-CONCEPT-V2.md`.

::note
**Einfachheit ist Leitprinzip**: Die Standardansicht zeigt wenige Entscheidungen — kein Slot- oder Regler-Zoo. Erweiterte Optionen stecken je Box hinter genau einem „Erweitert".
::

## Theme-Studio

- **Galerie:** `/dashboard/themes` (Zweispalten-Layout).
- **Editor:** Vollseite (`/new`, `/:id`) mit Dock aus zwei Boxen — „Farben" und „Schriften", je ein „Erweitert".
- **Schriften-Verwaltung:** `/dashboard/themes/fonts`.

## Custom Themes

- Table `custom_themes` (system-Migrationen 009–013).
- Die komplette Farb-Ramp entsteht **zur Laufzeit aus einer Basisfarbe** (`themes/shared/ramp.ts`, OKLCH, mit Tests).
- Das `config`-JSON wird **nur additiv** erweitert (kein `version`-Feld): `neutral: 'tinted'`, `font`/`fontHeading`, `darkAlias`, `headingWeight`/`Tracking`/`Uppercase`, `radius`.

## `<html>`-Attribute

Das theme-Plugin setzt die Attribute SSR-seitig in den Head (flash-frei): `data-theme` (`c-<rowId>`), `data-variant`, `data-neutral`, `data-font`, `data-font-heading`. Die Draft-Vorschau im Editor setzt sie direkt und stellt beim Verlassen den LIVE-Zustand aus `useTheme()` wieder her.

## Schriften: zwei Rollen

Text + Überschriften, dazu eine fixe Mono — **nie mehr als 3 Familien**:

- **Registry-Familien:** Einzelfamilien in `app/assets/css/fonts.css` (build-prozessiert, `@nuxt/fonts` self-hostet — nie nach `public/`).
- **Uploads:** WOFF2 in den Bucket `fonts` (Magic-Bytes-Check), Id `cf-<rowId>`, `@font-face` zur Laufzeit im Head.
- Legacy-Paar-Ids (z. B. `editorial`) mappt `resolveThemeFonts()`.

## Live-Propagation

`custom_themes`, `custom_fonts` und `app_config` sind Table-`read(any)` → das `realtime-themes`-Plugin refetcht debounct, der Head reagiert. Offene Fenster — auch von Gästen — **morphen ohne Reload**.

## Gut zu wissen

- Injizierte Theme-Styles sind **unlayered** und schlagen Tailwind-`@layer`-Utilities bewusst (z. B. `headingWeight` vs. `font-bold`).
- Visual-Tests zielen auf die deterministische `/visual`-Seite — nie Live-Daten screenshotten.
