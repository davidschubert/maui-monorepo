Der `admin`-Layer stellt das Dashboard-Gerüst — die Inhalte kommen aus den Feature-Layern über einen expliziten Vertrag.

## Modul-Registry: `maui.admin.modules`

Feature Layer registrieren ihre Admin-Seiten in der Registry (per `app.config.ts`-Merge), statt dass `admin` sie kennt:

- Einträge können `children` haben (Unterpunkte, RBAC-gefiltert; `exact` für Index-Einträge).
- Die Navigation im Dashboard-Layout entsteht vollständig aus der Registry.

## Feature-Katalog & Laufzeit-Toggles

Seit M2 gibt es eine Feature-Katalog-Seite im Admin:

- Katalog-Texte kommen aus den [Feature-Manifesten](/architektur/manifeste).
- Features lassen sich zur **Laufzeit** togglen (Persistenz in `app_config`).
- Grundgerüst-Features zeigen „**Immer aktiv**" statt eines toten Schalters.

## Weitere Admin-Bausteine

| Baustein | Beschreibung |
| --- | --- |
| Globale Suche | `server/api/admin/search.get.ts` — durchsucht die Admin-Welten |
| Changelog-Workflow | `pnpm changelog:draft` erzeugt Entwürfe aus Conventional Commits (seit erstem Tag via `--since`); öffentlicher Changelog ist microgecacht, Write invalidiert |
| Config-Seite | Laufzeit-Overrides in `app_config`, z. B. `aiModel` / `ticketsAiModel` (system-016) |
| Moderations-Queue | Meldungen, Assist, Bulk — über die `moderation`-Verträge |

## RBAC

Rollen & Rechte sind in `docs/RBAC-CONCEPT.md` beschrieben. Zwei Praxis-Regeln:

::tip
Permission-Flows immer auch mit einem **least-privileged User** (ohne Labels) testen — fremde `Role.label`-Permissions lassen sich über den Admin-Client setzen.
::

- Admin-Navigation und `children` werden RBAC-gefiltert — was der User nicht darf, erscheint nicht.
- Server-Routen prüfen die Berechtigung **immer selbst** (UI-Filterung ist nur Komfort).
