## Voraussetzungen

- **Node 22** (`nvm use 22`) und **pnpm** (Version siehe `packageManager` in der Root-`package.json`)
- **Appwrite 1.9.5 self-hosted** — lokal z. B. via Docker/OrbStack; Healthcheck:

```bash
curl http://localhost/v1/health/version
```

::warning
Für den Healthcheck **nie** `appwrite doctor` verwenden — das Kommando verschickt pro Lauf eine Test-SMTP-Mail.
::

## Installation

```bash
git clone git@github.com:davidschubert/maui-monorepo.git
cd maui-monorepo
nvm use 22
pnpm install
```

## App konfigurieren

Jede App hat eine eigene Appwrite-Instanz/-Projekt und liest die Konfiguration aus `.env` (Vorlage: `.env.example` in der jeweiligen App):

```bash [apps/comments/.env]
NUXT_PUBLIC_APPWRITE_ENDPOINT=http://localhost/v1
NUXT_PUBLIC_APPWRITE_PROJECT_ID=...
NUXT_APPWRITE_KEY=...            # Runtime-Key, server-only
```

Es gibt **zwei API-Keys pro Instanz** mit getrennten Scopes:

| Key | Scope | Verwendung |
| --- | --- | --- |
| Runtime-Key | sessions, users, rows, health | App zur Laufzeit (`.env`) |
| Migrations-Key | databases, tables, columns, indexes | nur Migrations-Scripts |

## Datenbank aufsetzen

Migrationen laufen **immer** über den zentralen Runner — bei mehreren Apps ist `--app` Pflicht:

```bash
pnpm migrate --app comments
```

Der Runner ist idempotent (409 → skip) und führt die Migrationen aller Layer in der korrekten Reihenfolge aus. Details: [Migrationen](/architektur/migrationen).

## Entwickeln

```bash
pnpm dev:core     # Core-Playground  → http://localhost:3000
pnpm dev:app      # comments  → http://localhost:3001
pnpm dev:docs     # diese Docs       → http://localhost:4000
```

| App | Port |
| --- | --- |
| `packages/core/.playground` | 3000 |
| `apps/comments` | 3001 |
| weitere Apps | 3002+ |
| `docs/` (diese Seite) | 4000 |

## Qualitäts-Checks

```bash
pnpm lint             # ESLint (geteilte Flat Config im Root)
pnpm typecheck        # vue-tsc pro Package
pnpm -r test          # Unit-Tests (Vitest)
pnpm check:manifests  # Manifest-Konsistenz (läuft auch in der CI)
```

::tip
Arbeitsprinzip im Projekt: **erst grün, dann weiter** — Fehler beheben und `lint`/`typecheck` grün haben, bevor der nächste Schritt gebaut wird.
::
