Das **Maui Monorepo** ist die gemeinsame Basis für alle Maui-Projekte: ein Nuxt-4-Monorepo, in dem Auth, Appwrite-Integration, Design-Fundament, Realtime und Utilities **einmal** implementiert und per `extends` in beliebig viele Apps eingebunden werden.

## Der Kern-Gedanke

Statt für jede Website denselben Unterbau neu zu bauen, gibt es drei Ebenen:

::card-group
  :::card
  ---
  icon: i-lucide-anchor
  title: Fundament-Layer
  ---
  `packages/core` (Auth, Appwrite-Clients, UI-Fundament, Utilities) und `packages/system` (Infra-Tabellen wie `audit_logs`, `app_config`, `notifications`). Core besitzt **keine** eigenen Appwrite Tables.
  :::

  :::card
  ---
  icon: i-lucide-package
  title: Feature Layers
  ---
  `packages/*` — abgeschlossene Features mit eigenem Datenmodell und/oder eigener UI-Welt: `themes`, `comments`, `posts`, `admin`, `billing`, `tickets`, `moderation` u. v. m.
  :::

  :::card
  ---
  icon: i-lucide-app-window
  title: Apps
  ---
  `apps/*` — dünne Kompositionen. Eine App wählt Features per Site-Manifest, bindet sie via `extends` ein und überschreibt nur, was site-spezifisch ist.
  :::
::

```ts [apps/<app>/nuxt.config.ts]
export default defineNuxtConfig({
  extends: [
    '../../packages/comments', // früher gelistet = höhere Priorität
    '../../packages/core',
    '../../packages/system',
  ],
})
```

::note
Früher gelistete Layer haben **höhere Priorität** — die App überschreibt alles, Feature Layer überschreiben das Fundament.
::

## Tech-Stack

| Baustein | Wahl |
| --- | --- |
| Framework | Nuxt 4.4.x (Composition API, SSR) |
| UI | Nuxt UI 4.x, Tailwind CSS 4 |
| State | Pinia (Composition-Style Stores) |
| Backend | Appwrite self-hosted 1.9.5 — TablesDB, Auth, Storage, Realtime, Presences |
| SDKs | `node-appwrite` (Server), `appwrite` (Web SDK, **nur** für Realtime) |
| Validierung | Zod (Schemas als `create*Schema(t)`-Factories) |
| i18n | `@nuxtjs/i18n` (de + en, `prefix_except_default`) |
| Tooling | pnpm Workspaces + Catalog, TypeScript strict, Node 22 |

## Strategie: drei Horizonte

Die Plattform-Strategie (siehe `docs/plans/MULTI-SITE-PLATTFORM-STRATEGIE.md` im Repo) denkt in drei Ausbaustufen:

1. **Eigene Sites** — mehrere eigene Projekte auf derselben Basis betreiben.
2. **Agentur** — Sites für Dritte aufsetzen und betreiben (Projekt-pro-Site in Appwrite).
3. **SaaS** — selbstbedienbare Plattform mit Feature-Wahl pro Site.

Jede App bekommt dabei ihre **eigene Appwrite-Instanz bzw. ihr eigenes Projekt** — Konfiguration ausschließlich über `.env`.
