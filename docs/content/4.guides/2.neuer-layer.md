## Checkliste

1. **Package anlegen:** `packages/<layer>/` mit `nuxt.config.ts`, `package.json` (`@maui/<layer>`, Versionen aus dem Catalog) und der [Standard-Struktur](/erste-schritte/struktur#aufbau-eines-layers).
2. **Feature-Manifest:** `feature.manifest.ts` mit `key` = Ordnername, `tier`, `requires`, Katalog-Texten — **nur `import type`**.
3. **Besitz klären:** Welche Tables, Routen, UI gehören diesem Layer? Bei Berührungspunkten mit anderen Layern → expliziten Vertrag definieren ([Layer-Modell](/architektur/layer-modell)).
4. **Migrationen** (falls eigene Tables): `scripts/migrations/`, idempotent + additiv, `hasMigrations: true`, Eintrag in `LAYER_ORDER` von `scripts/migrate.mjs` — siehe [Migrationen](/architektur/migrationen).
5. **Admin-Seiten** in `maui.admin.modules` registrieren (statt den admin-Layer anzufassen).
6. **GDPR-Contributor** registrieren, sobald der Layer User-Daten hält (`registerUserDataContributor` per Nitro-Plugin `server/plugins/user-data.ts`) — Pflicht.
7. **i18n:** `i18n/locales/de.json` + `en.json`; alle User-facing Strings über Keys.
8. **Pinia-Stores** via `imports.dirs` in der Layer-`nuxt.config.ts` registrieren.
9. `pnpm check:manifests` + `pnpm lint` + `pnpm typecheck` → grün.
10. In einer App testen: Feature ins Site-Manifest, `extends` + Dependency ergänzen, App starten.

## Stolperfallen

::warning
Die häufigsten Fehler beim Layer-Bau:
::

- `app.config.ts` in den Package-Root gelegt → wird stillschweigend ignoriert; sie gehört nach `app/`.
- Absolute Imports (`~/`, `@/`) im Layer → brechen, sobald die App den Layer extended. Immer relative Pfade.
- Domain-Types in `app/types/` → der Server sieht sie nicht; nach `shared/types/`.
- `useToast` im Core re-exportiert → schattet den Nuxt-UI-Auto-Import; kommt direkt aus Nuxt UI.
- Server-Plugins gelöscht/angelegt, während der Dev-Server lief → Neustart nötig; alte Compile-Fehler können im Log hängen bleiben (Wahrheit per `/_nuxt/@fs/<pfad>` prüfen).
- Neue Rows ohne explizite Permissions angelegt, obwohl der Inhalt moderierbar sein soll → Row-Level-Permissions von Anfang an mitdenken.

## Fundament vs. Feature

Bevor du baust, prüfe die Ebene: Braucht **jede** App die Fähigkeit (→ core/system), ist es eine Moderations-/Infra-Fähigkeit, an die Features andocken (→ moderation/system), oder eine abgeschlossene Feature-Welt (→ eigener Layer)? Fundament-Layer dürfen **nie** von Features abhängen.
