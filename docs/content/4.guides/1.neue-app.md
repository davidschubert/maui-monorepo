## `pnpm create-site`

Statt 5–10 Handschritten legt `scripts/create-site.mjs` eine neue Site in **einem** Befehl an (M4/P1):

```bash
pnpm create-site <name> [--features comments,posts,…] \
                        [--endpoint http://localhost/v1] [--port 30xx] \
                        [--skip-appwrite]
```

Das Script erledigt:

1. **Scaffold** aus `apps/_template` (Site-Manifest, `extends`, `package.json`, `.env.example`).
2. **Appwrite-Projekt + Keys + Platform** per Console-REST — dafür müssen `APPWRITE_CONSOLE_EMAIL` / `APPWRITE_CONSOLE_PASSWORD` als Env-Vars gesetzt sein. Ohne beide läuft nur der Scaffold plus eine manuelle Checkliste (`--skip-appwrite` implizit).
3. **.env** schreiben, **Install**, **Bootstrap** (DB/Buckets + manifest-gefilterte Migrationen).

Governance (F6): Die Projekt-ID ist `<name>-<shortid>` (unveränderlich, Slug getrennt); Key-IDs sind global eindeutig (`runtime-<name>`, …).

## Manuelle Nacharbeiten

- Port prüfen/vergeben (3002+, siehe [Setup](/erste-schritte/setup)).
- `pnpm check:manifests` — muss grün sein (Features, `extends`-Reihenfolge, `@maui/*`-Deps).
- App starten und den Auth-Flow einmal durchklicken.

## Feature nachträglich hinzufügen

1. Feature in `site.manifest.ts` ergänzen (inkl. transitiv benötigter `requires`).
2. `extends` in `nuxt.config.ts` und `@maui/*`-Dependency in `package.json` ergänzen — die kanonische Reihenfolge diktiert `check:manifests`.
3. `pnpm install`, dann `pnpm migrate --app <name>` (falls der Layer Tables hat).

::tip
Wenn du unsicher bist, wie die `extends`-Reihenfolge aussehen muss: einfach `pnpm check:manifests` laufen lassen — die Fehlermeldung nennt die erwartete Liste wörtlich.
::
