## Der zentrale Runner

Schema-Änderungen laufen **immer** über den zentralen Runner — nie Layer-Scripts direkt mit hartem env-file aufrufen:

```bash
pnpm migrate --app <app>
```

Bei mehreren Apps ist `--app` **Pflicht** — sonst landet die Migration auf der falschen Appwrite-Instanz. Der Runner führt die Migrationen aller Layer in der `LAYER_ORDER` (in `scripts/migrate.mjs`) aus; `check:manifests` hält diese Liste synchron mit den `hasMigrations`-Flags der Manifeste.

## Regeln für Migrations-Scripts

Migrationen liegen pro Layer unter `packages/<layer>/scripts/migrations/` und sind:

1. **Idempotent** — HTTP 409 („existiert schon") wird übersprungen; jede Migration kann beliebig oft laufen.
2. **Additiv-sicher per Default** — zerstörerische Aufrufe (`deleteTable`, `deleteColumn`, `deleteIndex`, `deleteRow(s)`, `update*Column`) sind nur mit begründetem Marker erlaubt:

```ts
// destruktiv-ok: Spalte war nie produktiv befüllt, Guard prüft rowCount === 0
await tablesDB.deleteColumn(/* … */)
```

Ohne Marker schlägt `pnpm check:manifests` fehl (M3-Migrations-Audit).

3. **Column-Polling** — nach dem Anlegen von Columns auf Status `available` pollen, **bevor** Indizes darauf erstellt werden (Appwrite legt Columns asynchron an).

## MariaDB-Grenzen im Blick

Appwrite läuft auf MariaDB mit utf8mb4 (4 Bytes/Zeichen):

- Varchar-Spalten max. **16.381** Zeichen, Zeilenbudget insgesamt ~65 KB.
- Große Textspalten deshalb ≤ ~15k Zeichen planen.

## Neuer Layer mit Tables — Checkliste

1. `scripts/migrations/NNN-*.ts` im Layer anlegen (idempotent, additiv).
2. `hasMigrations: true` im Feature-Manifest setzen.
3. Layer in die `LAYER_ORDER` von `scripts/migrate.mjs` eintragen.
4. `pnpm check:manifests` → grün.
5. `pnpm migrate --app <app>` gegen die Ziel-Instanz.
