Die Plattform-Strategie (F1) beendet die Doppelpflege von `extends`, `package.json` und Migrations-Reihenfolge: **Manifeste sind die Single Source of Truth**, ein CI-Check erzwingt Konsistenz.

## Feature-Manifest (pro Layer)

Jeder Layer unter `packages/` hat ein `feature.manifest.ts`:

```ts [packages/comments/feature.manifest.ts]
import type { FeatureManifest } from '../core/shared/types/manifest'

export default {
  key: 'comments',        // === Ordnername
  tier: 'feature',
  requires: ['posts'],    // andere Layer, die dieser Layer voraussetzt
  hasMigrations: true,    // ⇔ scripts/migrations/ existiert
  // + Katalog-Texte für die Feature-Auswahl im Admin
} satisfies FeatureManifest
```

::warning
Manifeste dürfen **nur `import type`** verwenden — sie werden mit `--experimental-strip-types` geladen, Wert-Importe brechen das.
::

## Site-Manifest (pro App)

Jede App unter `apps/` hat ein `site.manifest.ts` — die Single Source der Feature-Wahl:

```ts [apps/reddit-comments/site.manifest.ts]
export default {
  siteId: 'reddit-comments',   // === App-Ordnername
  features: ['themes', 'admin', 'comments', 'posts' /* … */],
  // core + system sind implizit immer dabei — NICHT listen
}
```

## Der Check: `pnpm check:manifests`

`scripts/check-manifests.mjs` läuft in der CI (lint) und lokal. Er prüft:

1. Jeder Layer hat ein Zod-valides Feature-Manifest, `key` = Ordnername, nur `import type`.
2. `hasMigrations` ⇔ `scripts/migrations/` existiert; jeder Layer mit Migrationen steht in der `LAYER_ORDER` von `scripts/migrate.mjs` (Drift-Erkennung in beide Richtungen).
3. Jede App hat ein Zod-valides Site-Manifest; Features existieren; `requires` ist **transitiv** erfüllt.
4. `extends` in der `nuxt.config.ts` = Features in kanonischer Reihenfolge (`EXTENDS_ORDER`) + `core` + `system` am Ende.
5. `@maui/*`-Dependencies in der `package.json` = exakt Features + `core` + `system`.

Zusätzlich enthält der Check den **Destruktiv-Guard** für Migrationen (siehe [Migrationen](/architektur/migrationen)).

::tip
Neue Layer und Apps **immer mit Manifest** anlegen — sonst schlägt die CI fehl. Der bequemste Weg für Apps ist [`pnpm create-site`](/guides/neue-app).
::

## Laufzeit-Gates

Die Feature-Wahl aus dem Manifest bestimmt, was gebaut wird. Zur **Laufzeit** kommen Config-Gates dazu (`app.config.ts`, Namespace `maui.*`, tief gemergt) plus Admin-Laufzeit-Toggles über `app_config` — Details unter [Core-Bausteine](/features/core-bausteine).
