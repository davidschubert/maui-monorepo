## Drei Ebenen

```
packages/core    ← Ebene 1: Fundament (besitzt KEINE Appwrite Tables!)
packages/system  ← Fundament: Infra-Tabellen (audit_logs, app_config, notifications)
packages/*       ← Ebene 2: Feature Layers (eigenes Datenmodell und/oder eigene UI-Welt)
apps/*           ← Ebene 3: dünne Apps, komponieren via extends
```

Die App listet ihre Layer in `extends` — **früher gelistet = höhere Priorität**. Die App überschreibt damit alles, Feature Layer überschreiben das Fundament. `core` und `system` stehen immer am Ende.

## Besitzverhältnisse (Layer-Grenzen)

Wer welche Tables, Routen und UI besitzen darf, regelt die **Layer-Grenzen-Matrix** in `docs/CONCEPT.md` (Abschnitt A14). Die wichtigsten Grundsätze:

- **Fundament-Layer (`core`, `moderation`, `system`) hängen nie von Features ab.** Die Abhängigkeitsrichtung zeigt immer von Feature → Fundament.
- **Cross-Layer-Abhängigkeiten sind explizite Verträge** — z. B. eine Registry-Funktion, die der eine Layer anbietet und der andere per Nitro-Plugin befüllt. Kein implizites Auto-Import- oder tableId-String-Coupling.
- **ESLint als Backstop:** `no-restricted-imports`-Regeln in der Root-Flat-Config verhindern künftige explizite Grenzverletzungen (z. B. darf `themes` als rein visueller Layer weder Appwrite noch andere Layer importieren).

## Beispiele für explizite Verträge

| Vertrag | Anbieter | Konsument |
| --- | --- | --- |
| `maui.admin.modules` (Nav-Registry) | admin | Feature Layer registrieren ihre Dashboard-Seiten |
| `registerUserDataContributor()` (GDPR) | core | jeder Layer mit User-Daten |
| `registerReportEscalationHandler()` | moderation | comments (Auto-Hide), posts |
| `resolveReportsForTarget()` / `openReportsForTarget()` | moderation | Resolve/Assist/Bulk in Admin-Flows |
| `registerEmbeddableRoute()` (frame-ancestors) | core | comments (Embed) |

::note
Faustregel bei neuen Abhängigkeiten: erst fragen, **wem** die Fähigkeit gehört (Besitz-Matrix), dann einen expliziten Vertrag definieren — nie einfach über Auto-Import „rüberlangen".
::

## Was gehört wohin?

- **Neue Tabelle?** → gehört einem Feature Layer (oder `system` für Infra). Core bleibt table-frei.
- **Neue Admin-Seite?** → lebt im Feature Layer, registriert sich in `maui.admin.modules` (mit optionalen `children`, RBAC-gefiltert).
- **Geteilte UI/Utility ohne Datenmodell?** → core.
- **Moderations-Logik?** → Zweiphasen-Hide + Cascade gehören dem `comments`-Layer (`commentModeration.ts`); `moderation` zählt Meldungen und eskaliert, der Owner-Layer reagiert.
