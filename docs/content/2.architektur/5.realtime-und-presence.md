## Ein geteilter SDK-Socket

Seit P1 (2026-07) läuft die gesamte Realtime-Kommunikation über **eine** geteilte, JWT-authentifizierte Web-SDK-Verbindung in `core/app/composables/useRealtimeClient.ts` (`sharedRealtime`, `realtimeCookieClient`, `ensureRealtimeJwt`):

- `useRealtimeRows`, Presence und Config-Flags **multiplexen über denselben Socket** (`Channel.tablesdb().table().row()`, optional mit server-seitigen `queries`; der clientseitige `where`-Filter bleibt Sicherheitsnetz).
- Das JWT kommt von `GET /api/auth/realtime-token` (15 min gültig, der Client refresht selbst).

::warning
Cookie-Client **nie** mit JWT mischen — das quittiert Appwrite mit einem 403.
::

::note
**Bewusste Ausnahme:** `useRealtimeAccount` bleibt ein cookie-nativer WebSocket — Instant-Session-Revoke hängt am Cookie-Close. Nicht konsolidieren.
::

Ops-Hinweis: Realtime braucht einen gesunden `appwrite-realtime`-Container. Bei Swoole-Crash:

```bash
docker compose up -d --no-deps appwrite-realtime
```

## Presence: eine Presence pro User

Die gesamte Presence ist auf die Appwrite **Presences API** (self-hostbar seit 1.9.5) vereinheitlicht — **eine** Presence pro User (`presenceId = userId`), die `metadata` trägt den Kontext. Es gibt **keine** presence-Table mehr.

### Warum der Write server-seitig läuft

In der SSR-Cookie-Architektur hat der Web-SDK-Client im Browser keine Session — `upsertPresence()` über einen Guest-WS wird verworfen, `PUT /presences` liefert 401. Deshalb:

- **Write:** `POST /api/presence/heartbeat` upsertet mit dem Admin-Client (`read("users")`, `expiresAt` 90 s).
- **Autorität:** `usePresenceState()` ist die einzige Heartbeat-Quelle pro Tab (bei Login/metadata-Änderung, alle 20 s, bei `visibilitychange`/`focus`).
- **Read:** `usePresence(predicate)` liest direkt über die Presences-API (`presences.list()` per Cookie-GET funktioniert, `Channel.presences()` triggert Updates). „Online jetzt" = `updatedAt`-Recency 60 s.
- **Server:** `listOnlinePresences()` in `core/server/utils/presence.ts`.

### metadata-Felder

Jedes Feld hat einen eigenen Zweck, sie kollidieren nicht:

| Feld | Bedeutung | Use-Case |
| --- | --- | --- |
| `scope` | Thread-Kontext | `useThreadPresence` |
| `typing` | tippt gerade | `useThreadPresence` |
| `replyingTo` | antwortet auf Kommentar | `useThreadPresence` |
| `near` | Lese-Position (commentId) | `useThreadPresence` |
| `action` | `reviewing:*` / `editing:*` | `useModerationPresence`, `useEditAwareness` |
| `page` | aktuelle Dashboard-Seite | `useViewingPresence` → „N sehen diese Seite" |

`PresenceAvatar` (core) rendert Avatar + Icon-Badge in der Ecke (tippt/antwortet).

## Live-Propagation der Themes

`custom_themes`, `custom_fonts` und `app_config` sind Table-`read(any)` — das `realtime-themes`-Plugin refetcht debounct und der Head reagiert: offene Fenster (auch Gäste) morphen ohne Reload. Details: [Themes](/features/themes).
